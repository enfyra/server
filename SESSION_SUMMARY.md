# Session Summary — MySQL Support & Query Builder Rewrite

## Starting Point

Enfyra server had PostgreSQL-only query builder. MySQL support needed for multi-DB consistency.

## Changes Made

### 1. Batch-Fetch Query Architecture (Major Rewrite)

**Problem**: Query builder embedded JSON aggregation subqueries (json_agg/JSON_ARRAYAGG) directly in SQL. This required DB-specific code paths, MySQL-specific hacks (derived tables, GROUP_CONCAT, sort_buffer_size tuning), and crashed on large result sets (46K rows/group = sort buffer overflow).

**Solution**: Replaced single-query JSON aggregation with batch-fetch approach — separate simple queries per relation type, assembled in Node.js.

**Files changed**:
- `src/infrastructure/query-builder/utils/sql/batch-relation-fetcher.ts` — NEW: core batch-fetch utility
- `src/infrastructure/query-builder/utils/sql/expand-fields.ts` — REWRITTEN: returns BatchFetchDescriptors instead of CTE/subquery SQL
- `src/infrastructure/query-builder/executors/sql-query-executor.ts` — MODIFIED: calls executeBatchFetches after main query, before parseResult
- `src/infrastructure/query-builder/utils/sql/nested-subquery-builder.ts` — DELETED: no longer needed

**How batch-fetch works**:
```
Main query: SELECT scalar columns + FK columns FROM table WHERE ... ORDER BY ... LIMIT N
  → parentIds extracted

Per relation type (one query each):
  M2O: SELECT * FROM target WHERE id IN (...fkValues)
  O2M: SELECT * FROM child WHERE fkCol IN (...parentIds) ORDER BY id
  M2M: SELECT j.sourceCol, t.* FROM junction j JOIN target t ON ... WHERE j.sourceCol IN (...parentIds) ORDER BY t.id

Node.js: group by parent ID, stitch into parent rows
Recursive: nested relations batch-fetched at each level (up to maxDepth)
```

**Key design decisions**:
- Zero DB-specific SQL functions — works identically on MySQL, Postgres, SQLite
- FK columns included in main SELECT for M2O stitching, stripped after
- O2M FK deletion deferred until after nested batch-fetches (prevents FK consumed before nested M2O can use it)
- CTE still used for pagination anchor (limited_parent) — only the relation embedding was replaced

**Benchmark results (1M posts, MySQL 8.4)**:
| Query | Time |
|---|---|
| Posts no-deep (limit=50) | 30.7ms |
| Posts + M2O (author+category) | 31.1ms |
| Posts + M2M (tags) | 32.3ms |
| Posts + all relations | 30.8ms |
| Authors + O2M posts (10 authors) | 3.5ms |
| Authors + nested posts.tags | 3.7ms |
| Tags + M2M inverse (46K posts/tag) | 2.2s (was CRASH before) |

### 2. Cascade Transaction Atomicity

**Problem**: `qb.insert` and `qb.update` in KnexService ran cascade operations (M2M junction sync, O2M FK updates, nested creates) without wrapping in a transaction. If cascade step N failed, steps 1..N-1 were already auto-committed → data inconsistency. Only `qb.delete` was properly wrapped.

**Solution**: All three operations (insert, update, delete) now use shared `ensureTransaction()` helper. If already in a transaction context → reuse. If not → create new transaction wrapping the entire operation + cascade hooks.

**File**: `src/infrastructure/knex/knex.service.ts` — `wrapQueryBuilder()` method, `ensureTransaction()` helper

### 3. Worker Abort Signal for Transaction Cleanup

**Problem**: When executor worker timed out (30s) or crashed, any Knex transaction opened on the main thread via IO bridge was orphaned — never committed or rolled back. This held locks on MySQL indefinitely, blocking all subsequent requests to the affected tables.

**Solution**: AbortController per executor task. On timeout/crash → abort signal fires → propagated via AsyncLocalStorage to any code running in that IO context → active transactions rollback immediately.

**Files**:
- `src/infrastructure/executor-engine/services/isolated-executor.service.ts` — AbortController per task, abort on timeout/crash, ioAbortContext AsyncLocalStorage, `getIoAbortSignal()` export
- `src/infrastructure/knex/knex.service.ts` — `ensureTransaction()` listens to abort signal, rollbacks on abort
- `src/modules/table-management/services/sql-table-handler.service.ts` — createTableInternal, updateTable, deleteTableInternal all listen to abort signal on their explicit transactions

### 4. Schema Migration Self-Deadlock Fix

**Problem**: `SqlSchemaMigrationService.createTable()` called `updateMetadataIndexes()` at the end, which did `knex('table_definition').update(...)` using a pool connection. But the caller (`createTableInternal`) still held an uncommitted transaction on that same row → self-deadlock → lock wait timeout → worker crash.

**Solution**: `updateMetadataIndexes` moved out of `createTable()`, called after `trx.commit()` in the handler. Same for `updateTable()`'s `updateMetadataFields`.

**Files**:
- `src/infrastructure/knex/services/sql-schema-migration.service.ts` — createTable returns autoGeneratedIndexes, updateTable returns pendingMetadataUpdate, new public methods `updateMetadataIndexesForTable()` and `applyPendingMetadataUpdate()`
- `src/modules/table-management/services/sql-table-handler.service.ts` — calls metadata update after trx.commit()

### 5. Bootstrap Index Sync — FK Index Skip

**Problem**: MySQL auto-creates indexes on FK columns. Bootstrap index sync tried to drop them (not in snapshot) → "Cannot drop index needed by foreign key constraint" warnings on every restart.

**Solution**: When comparing indexes to drop, skip single-column indexes that back a FK constraint.

**File**: `scripts/utils/sql/schema-comparison.ts` — fkColumnSet filter in indexesToRemove logic

### 6. Schema Migration — M2M FK Constraint Bug

**Problem**: `createTable()` in SqlSchemaMigrationService iterated ALL relations to create FK constraints, including M2M. M2M doesn't have a FK column on the source table → "Key column doesn't exist" error on both MySQL and Postgres.

**Solution**: Added relation type filter — only create FK constraints for many-to-one and one-to-one relations.

**File**: `src/infrastructure/knex/services/sql-schema-migration.service.ts` — added `if (!['many-to-one', 'one-to-one'].includes(rel.type)) continue;` before FK creation loop

### 7. Snapshot Migration Fixes

**Files**: `data/snapshot.json`, `data/snapshot-migration.json`
- `package_definition.installedBy`: isNullable false → true (FK onDelete SET NULL requires nullable column)
- `relation_definition.targetTable`: isNullable false → true (same reason)
- Added corresponding entries in snapshot-migration.json for existing DB migration
